Sécurité en PHP : Les concepts clés pour l'épreuve

Maîtrisez les vulnérabilités courantes et leurs solutions en PHP pour réussir l'épreuve du BTS SIO.

1. CSRF (Cross-Site Request Forgery)

🔴 Le problème :

Un attaquant force un utilisateur authentifié à exécuter des actions non désirées sur une application web.

Exemple : Un lien caché dans un email qui change l'email de l'utilisateur sur un site vulnérable.

✅ La solution :

Utiliser des tokens CSRF uniques pour chaque session et les vérifier à chaque action sensible.

// Génération du token CSRF

session_start();

if (empty($_SESSION['csrf_token'])) {

$_SESSION['csrf_token'] = bin2hex(random_bytes(32));

}

// Dans le formulaire

<input type="hidden" name="csrf_token"

value="<?= $_SESSION['csrf_token'] ?>">

// Vérification à la réception

if (!hash_equals($_SESSION['csrf_token'], $_POST['csrf_token'])) {

die('Token CSRF invalide');

}

📝 À retenir pour l'épreuve :

  • Les tokens doivent être imprévisibles (random_bytes)
  • Vérifier avec hash_equals (pas == qui est vulnérable)
  • Les actions POST/DELETE/PUT doivent être protégées

2. XSS (Cross-Site Scripting)

🔴 Le problème :

Injection de scripts malveillants dans des pages web consultées par d'autres utilisateurs.

Exemple : <script>alert('Hacké!')</script> dans un commentaire.

❌ Code vulnérable 

<?php
echo "Bonjour " . $_GET['nom'];
?>

✅ Code sécurisé 

<?php
echo "Bonjour " . htmlspecialchars($_GET['nom'], ENT_QUOTES, 'UTF-8');
?>

✅ Les solutions :

1. Échappement systématique :

// Échapper à l'affichage
echo htmlspecialchars($data, ENT_QUOTES, 'UTF-8');

// Pour les attributs HTML
echo '<input value="' . htmlspecialchars($value, ENT_QUOTES) . '">';

2. Content Security Policy (CSP) : 

header("Content-Security-Policy: default-src 'self'");

3. Modifications de BDD (Injections SQL)

🔴 Le problème :

Injection de code SQL malveillant dans les requêtes pour modifier, voler ou supprimer des données.

❌ Requête vulnérable (concaténation directe) 

$id = $_GET['id'];
$sql = "SELECT * FROM users WHERE id = " . $id;
// Attaque possible : ?id=1 OR 1=1 UNION SELECT * FROM passwords

✅ Requête sécurisée avec requêtes préparées (PDO) 

try {
    $pdo = new PDO("mysql:host=localhost;dbname=test;charset=utf8", 
                   "user", "pass");
    $pdo->setAttribute(PDO::ATTR_ERRMODE, PDO::ERRMODE_EXCEPTION);
    
    // Requête préparée
    $stmt = $pdo->prepare("SELECT * FROM users WHERE id = :id");
    $stmt->execute(['id' => $_GET['id']]);
    $result = $stmt->fetch();
    
} catch(PDOException $e) {
    error_log($e->getMessage()); // Jamais d'affichage direct
    die("Erreur de base de données");
}

📊 Types d'attaques SQL à connaître :

  • UNION-based : Récupérer des données d'autres tables
  • Blind SQL : Deviner les données vrai/faux
  • Time-based : Mesurer les délais de réponse
  • Error-based : Exploiter les messages d'erreur

🔐 Bonnes pratiques :

  • Toujours utiliser des requêtes préparées
  • Jamais de concaténation de variables dans les requêtes
  • Valider et filtrer les entrées (filter_var, ctype_digit...)
  • Principe du moindre privilège pour l'utilisateur BDD
  • Ne pas afficher les erreurs SQL en production

4. Contrôleurs et Architecture MVC

🏗️ Structure MVC expliquée :

  • Modèle (Model) : Gère les données et la logique métier
  • Vue (View) : Affiche l'interface utilisateur
  • Contrôleur (Controller) : Traite les requêtes et coordonne

Exemple de contrôleur sécurisé : 

class UserController {
    private $userModel;
    
    public function __construct() {
        $this->userModel = new UserModel();
    }
    
    // Action avec validation et protection
    public function updateProfile($request) {
        // 1. Vérifier l'authentification
        if (!isset($_SESSION['user_id'])) {
            header('Location: /login');
            exit;
        }
        
        // 2. Vérifier le token CSRF
        if (!isset($request['csrf_token']) || 
            !hash_equals($_SESSION['csrf_token'], $request['csrf_token'])) {
            die('Invalid CSRF token');
        }
        
        // 3. Valider les données
        $errors = [];
        
        if (empty($request['email'])) {
            $errors[] = "L'email est requis";
        } elseif (!filter_var($request['email'], FILTER_VALIDATE_EMAIL)) {
            $errors[] = "Email invalide";
        }
        
        if (!empty($errors)) {
            // Afficher les erreurs à l'utilisateur
            return $this->view('profile/edit', ['errors' => $errors]);
        }
        
        // 4. Nettoyer et mettre à jour
        $data = [
            'email' => htmlspecialchars($request['email'], ENT_QUOTES),
            'name' => htmlspecialchars($request['name'], ENT_QUOTES)
        ];
        
        // 5. Utiliser le modèle (avec requête préparée)
        if ($this->userModel->update($_SESSION['user_id'], $data)) {
            $_SESSION['success'] = "Profil mis à jour";
            header('Location: /profile');
            exit;
        }
    }
    
    private function view($view, $data = []) {
        extract($data);
        require "views/{$view}.php";
    }
}

🎯 Rôle du contrôleur dans la sécurité :

  • Point d'entrée unique pour valider les requêtes
  • Gestion de l'authentification et des sessions
  • Vérification des autorisations (rôles, permissions)
  • Validation et filtrage des données entrantes
  • Protection CSRF avant d'appeler le modèle

5. Hameçonnage (Phishing)

🎣 Qu'est-ce que le phishing ?

Technique d'ingénierie sociale où l'attaquant se fait passer pour une entité légitime pour voler des informations sensibles.

❌ Email frauduleux :

De : support@paypa1.com

Objet : Votre compte va être suspendu

Cliquez ici pour vérifier votre compte :

http://fake-paypal.com/login

⚠️ URL suspecte, fautes d'orthographe, urgence

✅ Comment reconnaître :

  • URL qui ne correspond pas au site officiel
  • Fautes d'orthographe et formulations étranges
  • Sensations d'urgence ou menaces
  • Demandes d'informations personnelles
  • Pièces jointes suspectes

🛡️ Protection côté développeur : 

// 1. Authentification à deux facteurs (2FA)
// 2. En-têtes anti-phishing
header("X-Frame-Options: DENY"); // Évite le clickjacking
header("X-Content-Type-Options: nosniff");
header("Referrer-Policy: strict-origin-when-cross-origin");

// 3. Formation des utilisateurs
// 4. Domaines vérifiés (SPF, DKIM, DMARC pour les emails)
// 5. Détection d'URLs similaires (typosquatting)

🔍 Vérifications pour l'utilisateur :

  • Vérifier l'URL dans la barre d'adresse
  • Regarder le cadenas HTTPS (mais ne suffit pas)
  • Ne jamais cliquer sur les liens dans les emails
  • Contacter l'entreprise par un autre moyen
  • Utiliser un gestionnaire de mots de passe (ils détectent les faux sites)

📋 Résumé des bonnes pratiques PHP

Contre les injections :

  • Requêtes préparées (PDO)
  • Validation des entrées
  • Échappement avec htmlspecialchars

Contre CSRF :

  • Tokens uniques par session
  • Vérification avec hash_equals
  • SameSite cookies

Architecture :

  • Séparation MVC
  • Contrôleurs = validateurs
  • Modèles = requêtes sécurisées

En-têtes HTTP :

  • Content-Security-Policy
  • X-Frame-Options: DENY
  • X-Content-Type-Options
